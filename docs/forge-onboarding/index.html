01 / MAPA

A fábrica é um loop, não uma sequência de prompts

Cada etapa tem entrada, dono, artefato e verificação. O modelo é só o motor daquela volta.

Agente executa Humano decide Evidência fica no repo
    ↺
    Depois do ship, o loop continua.

    Feedback volta por ITERATE. P4 mede comportamento real. P5 mata o que não paga ou escala o que provou sinal.

    02 / START

    Primeiros 15 minutos

    Aprenda com uma pipeline completa usando o executor fake. Mesmos artefatos, mesmos gates, zero quota.

    1. 01

      Instale o workspace

      Use Node 20 ou superior. Rode apenas quando dependências estiverem ausentes.

      npm install
    2. 02

      Abra o Maestro

      O helper sobe o server e abre a interface local. Se a página mostrar só “Running”, você acertou o Docker, não o Maestro.

      npm run maestro:open
    3. 03

      Defina a fábrica

      Nicho, stack, contexto e idiomas viram instruções persistentes para todos os jobs.

      npm run forge -- profile init
    4. 04

      Monte o time

      Escolha provedor, modelo, esforço e função sem editar o roster à mão.

      npm run forge -- team
    5. 05

      Rode sem gastar quota

      Seu ensaio geral. O fake produz artefatos substanciais e pausa no mesmo p0-go do run real.

      npm run forge -- new "minha ideia" --team dry-run --dry-run
    6. 06

      Veja o estado

      Confirme job, player, gate e histórico de cada pipeline.

      npm run forge -- status
    7. 07

      Decida com evidência

      Leia o scorecard e só então responda ao gate.

      npm run forge -- decide p0-go go --app meu-app

    Você chegou lá quando

    • o app ganhou um repositório próprio;
    • o scorecard contém seção Mercado;
    • o verify terminou PASS;
    • a pipeline parou em p0-go;
    • nenhum provedor real gastou quota.

    06 / OPERATOR

    Modo operador: confiança vem do estado

    O Forge aguenta paralelismo, falha e restart porque o run vive em arquivos, não na memória de uma conversa.

    N PIPELINES

    Cada app tem estado e git próprios

    Os engines compartilham apenas o que precisa ser global, como cooldown de provider. O working tree de um app não disputa com o outro.

    anima-deckP4 · measure doki-calldeploy · gate nova-ideiaL1/B3 · running

    FAIL ≤ 3

    Retry não é loop infinito

    Verify objetivo, até três correções, rollback via git e L2 quando quota ou provedor bloqueiam.

    COOLDOWN

    Rate-limit atravessa pipelines

    Um provider saturado fica inelegível globalmente. O estado sobrevive a restart e o fallback assume.

    07 / RECOVERY

    Algo parou. Qual comando usar?

    Comece pelo sintoma. O objetivo é preservar trabalho e reduzir blast radius.

    01
    Existe job vivo?

    Sim: pause antes de mexer no server.

    forge stop meu-app
    02A

    Código do Maestro mudou

    Recarregue. O restart recusa job vivo por padrão.

    forge restart
    02B

    Pipeline está parada

    Se não há gate pendente, continue do estado persistido.

    forge resume meu-app
    02C

    Provider saturou

    Não reinicie produto. Aguarde cooldown ou deixe fallback assumir.

    forge status
    03
    Run sem saída segura?

    kill encerra o run. Não apaga o app.

    forge kill meu-app

    ZONA DE RISCO · IRREVERSÍVEL

    remove não é um “kill mais forte”. Ele apaga.

    Remove o repo do app, estado da pipeline e propostas. Confirme o app-id, verifique se está em produção e só então execute.

    npm run forge -- remove meu-app --force
